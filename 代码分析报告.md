# NovelForge 代码分析报告

生成时间：2026-05-11

## 1. 执行摘要

NovelForge 是一个面向长篇小说创作的 AI 辅助写作系统。核心产品模型围绕项目、卡片、动态 Schema、上下文注入、知识图谱、指令流生成、工作流和 Agent 协作展开。当前代码整体呈现为：

- 后端：FastAPI + SQLModel + SQLite 为主，Neo4j 可选，LangChain 负责模型适配。
- 前端：Electron + Vue 3 + TypeScript + Pinia + Element Plus，同时支持 Web 模式。
- 工作流：已经从旧 DAG 思路切换到代码式工作流，后端负责 parse/validate/run，前端提供编辑、执行、进度展示。
- 状态联动：后端通过事件总线触发工作流，前端通过 `X-Workflows-Started` 响应头、SSE 和轮询刷新卡片。
- Web 部署：新增 Windows Web 一键部署脚本，前端通过运行时 `novelforge-config.js` 读取后端地址，支持同源 NPM 反代部署。
- 灵感助手：对话历史已从浏览器本地存储升级为后端 SQLite 持久化，同一后端下的不同浏览器可同步查看历史会话。

整体架构已经有明确分层，但存在几类后续修改前必须注意的风险：数据库迁移机制偏轻量、工作流运行控制大量依赖进程内状态、API Key 安全存储存在双轨、前端路由与流式网络层不够统一、Web 反代场景对 API 尾斜杠敏感、大组件和 Store 职责偏重。

## 2. 项目定位与核心领域模型

README 将 NovelForge 定位为“新一代 AI 长篇小说创作引擎”，并明确四个核心理念：卡片、动态输出模型、上下文注入、知识图谱，见 `README.md:27`、`README.md:29`。

主要领域对象如下：

- 卡片类型：定义默认 Schema、AI 参数、编辑器组件、单例约束和 UI 布局，见 `backend/app/db/models.py:77`。
- LLM 配置：保存 provider、model、api_base、api_key、配额统计等，见 `backend/app/db/models.py:18`。
- Prompt：内置或用户定义提示词模板，见 `backend/app/db/models.py:67`。

## 4. 后端架构分析

### 4.1 应用启动链路

后端启动流程：

1. `backend/main.py:4` 从可执行文件目录、后端目录、当前工作目录加载 `.env`。
2. `backend/main.py:31` 定义 lifespan。
3. `backend/main.py:34` 调用 `startup()`。
4. `backend/app/core/startup.py:188` 初始化数据库。
5. `backend/app/core/startup.py:190` 初始化内置数据。
6. `backend/app/core/startup.py:192` 注册事件处理器。
7. `backend/app/core/startup.py:194` 注册工作流节点。
8. `backend/app/core/startup.py:196` 清理重启后遗留的 running 工作流。
9. `backend/main.py:76` 挂载 API 路由。

启动设计的优点是职责清晰，bootstrap、事件和工作流节点都走自动发现。但数据库初始化仍处于“开发态可用、生产建议迁移”的状态，见 `backend/app/core/startup.py:18` 到 `backend/app/core/startup.py:27`。

### 4.2 API 路由边界

总路由在 `backend/app/api/router.py:15` 创建，主要挂载：

- `/projects`：项目管理，见 `backend/app/api/router.py:16`。
- `/llm-configs`：LLM 配置，见 `backend/app/api/router.py:17`。
- `/ai`：AI 生成与助手相关接口，见 `backend/app/api/router.py:19`、`backend/app/api/router.py:20`。
- `/prompts`：提示词，见 `backend/app/api/router.py:22`。
- 卡片接口挂在空 prefix 下，见 `backend/app/api/router.py:23`。
- `/chapter-reviews`：章节审核，见 `backend/app/api/router.py:24`。
- `/context`、`/memory`、`/relation-graph`、`/foreshadow`、`/knowledge`：上下文、记忆、关系图谱和知识库，见 `backend/app/api/router.py:26` 到 `backend/app/api/router.py:30`。
- workflows：工作流接口，见 `backend/app/api/router.py:31`。

后续新增 API 时，应先确认是否属于已有领域 service；避免把跨领域流程直接写进 endpoint。工程规则也要求跨域联动优先事件发布/订阅，见 `rules/novelforge-engineering-rule.md:27`。

灵感助手当前挂在 `/api/ai/assistant` 下，包含流式对话 `/chat` 和会话持久化 `/sessions`，见 `backend/app/api/endpoints/assistant.py:20`、`backend/app/api/endpoints/assistant.py:43`。Web 反代部署下应避免触发 FastAPI 自动 307 重定向；集合接口调用需与后端声明的尾斜杠保持一致，例如提示词列表使用 `/prompts/`，见 `frontend/src/renderer/src/api/setting.ts:73`。

- 知识图谱 provider：默认 `sqlmodel`，见 `backend/app/core/config.py:59`。
- Neo4j 连接：默认 `neo4j://127.0.0.1:7687`，见 `backend/app/core/config.py:71`。
- Bootstrap 覆盖：`BOOTSTRAP_OVERWRITE` 默认 `True`，见 `backend/app/core/config.py:103`。

- API 前缀：默认 `/api`，见 `backend/app/core/config.py:160`。

风险点：生产部署应重点收紧 CORS；默认 bootstrap 覆盖可能影响用户改过的内置数据；README、`.env.example`、代码里的知识图谱 provider 命名需要统一。

### 4.4 数据库与迁移

数据库 Session 在 `backend/app/db/session.py:8` 创建，SQLite 使用 `check_same_thread=False`，事务依赖在请求成功后 commit，异常 rollback，见 `backend/app/db/session.py:15` 到 `backend/app/db/session.py:28`。

启动时 `SQLModel.metadata.create_all(engine)` 创建表，并通过 `_ensure_safe_additive_columns()` 对已有表自动补安全追加列，见 `backend/app/core/startup.py:24`、`backend/app/core/startup.py:54`。代码明确说明不处理删列、改列类型、改约束、索引补建和数据回填，见 `backend/app/core/startup.py:57` 到 `backend/app/core/startup.py:60`。

近期新增的 `AssistantConversation` 是新表，依赖 `create_all` 在启动时创建；如果后续要调整消息结构、索引或约束，应补正式迁移而不是继续依赖轻量补列。

建议：后续只要涉及数据模型字段删除、类型变化、索引、唯一约束、外键变化，都应引入或补齐 Alembic 迁移，不应继续依赖启动补列。

### 4.5 事件总线

事件总线在 `backend/app/core/events.py`：

- `@on_event` 装饰器注册处理器，见 `backend/app/core/events.py:29`。
- `emit_event` 同步遍历处理器，见 `backend/app/core/events.py:52`。
- 单个处理器异常只记录日志，不向调用方传播，见 `backend/app/core/events.py:69` 到 `backend/app/core/events.py:73`。

这与工程规则“跨域联动优先事件发布/订阅”一致，但也意味着项目或卡片保存成功后，下游工作流可能静默失败。若后续修改的是关键联动，建议增加可观测性，例如结构化错误事件、运行记录或 UI 提示，而不是改变保存接口的成功语义。

### 4.6 AI 调用边界

后端 AI 能力主要集中在 `backend/app/services/ai/`：

- ChatModel 构建由 `chat_model_factory` 负责。
- 结构化生成、续写和流式生成由 `llm_service` 等服务负责。
- 工作流 Agent 走 `workflow_agent` 子模块。

LLM 配置在数据库模型中直接保存 `api_key`，见 `backend/app/db/models.py:24`。虽然前端 Electron 主进程存在 Keytar IPC，但当前后端模型仍以明文字段作为核心配置来源。后续涉及 API Key、安全存储、导入导出时必须先统一策略。

### 4.7 工作流系统

工作流核心数据在 `Workflow`、`WorkflowRun`、`NodeExecutionState`：

- 工作流代码作为 `definition_code` 保存，见 `backend/app/db/models.py:185`。
- 触发器缓存 `triggers_cache` 从代码中自动提取，代码是唯一真实来源，见 `backend/app/db/models.py:197` 到 `backend/app/db/models.py:217`。
- 运行状态写入 `WorkflowRun.status/state_json/error_json/summary_json`，见 `backend/app/db/models.py:235` 到 `backend/app/db/models.py:250`。
- 节点状态、错误和检查点写入 `NodeExecutionState`，见 `backend/app/db/models.py:274` 到 `backend/app/db/models.py:298`。

运行控制器 `WorkflowRuntime` 是进程内共享状态，包含并发槽、任务、执行器、取消和暂停请求，见 `backend/app/services/workflow/engine/runtime.py:28` 到 `backend/app/services/workflow/engine/runtime.py:40`。触发器防抖同样是进程内字典 `_recent_keys`，见 `backend/app/services/workflow/triggers.py:21` 到 `backend/app/services/workflow/triggers.py:24`。

风险点：

- 服务重启后，运行控制状态无法恢复，只能把 running 标记为 failed，见 `backend/app/core/startup.py:146` 到 `backend/app/core/startup.py:176`。
- 多进程或横向扩容时，进程内防抖、暂停、取消、并发槽都不一致。
- SQLite 并发写入和后台线程执行组合需要谨慎，见 `backend/app/db/session.py:8` 和 `backend/app/services/workflow/triggers.py:8` 到 `backend/app/services/workflow/triggers.py:18`。

工程规则要求任何工作流代码改动必须 parse、validate、通过后才应用，见 `rules/novelforge-engineering-rule.md:69` 到 `rules/novelforge-engineering-rule.md:77`。

### 5.3 API 层

HTTP 客户端位于 `frontend/src/renderer/src/api/request.ts`：

- Web 生产部署优先读取 `window.__NOVELFORGE_CONFIG__.backendBaseUrl`，见 `frontend/src/renderer/src/api/request.ts:28` 到 `frontend/src/renderer/src/api/request.ts:31`。

- Electron 默认 `http://127.0.0.1:54321`，见 `frontend/src/renderer/src/api/request.ts:48` 到 `frontend/src/renderer/src/api/request.ts:49`。
- `resolveRequestUrl()` 会将相对路径解析为绝对 URL，见 `frontend/src/renderer/src/api/request.ts:14` 到 `frontend/src/renderer/src/api/request.ts:17`。

- 普通请求和 AI 请求均使用 Axios fetch adapter，见 `frontend/src/renderer/src/api/request.ts:190`、`frontend/src/renderer/src/api/request.ts:197`。

当前网络层存在 axios、fetch、EventSource 并存的情况。`useCardStore` 内部直接 fetch 工作流运行状态和卡片详情，见 `frontend/src/renderer/src/stores/useCardStore.ts:170`、`frontend/src/renderer/src/stores/useCardStore.ts:198`。后续应统一 SSE、轮询、取消、错误提示和鉴权策略。Web 反代场景还必须关注 FastAPI 尾斜杠重定向，项目列表已改为 `/projects/`，见 `frontend/src/renderer/src/api/projects.ts:21`；提示词列表已改为 `/prompts/`，见 `frontend/src/renderer/src/api/setting.ts:73`。

## 7. 文档与代码不一致点

以下点建议后续优先澄清：

- README 表述 Neo4j 可选且默认 SQLite，但发行版运行步骤又提示确保 Neo4j Desktop 已启动。需要统一“默认是否需要 Neo4j”。
- `.env.example` 使用 `KNOWLEDGE_GRAPH_PROVIDER=sqlite`，代码默认 provider 是 `sqlmodel`，见 `backend/app/core/config.py:59`。需要确认二者是否等价，避免部署误配。
- 前端依赖包含 `vue-router`，但应用根部主要使用手写 hash 路由，见 `frontend/package.json:61` 和 `frontend/src/renderer/src/App.vue:52`。
- Electron 主进程提供 Keytar 安全存储 IPC，见 `frontend/src/main/index.ts:98`，但后端 LLM 配置模型仍明文保存 `api_key`，见 `backend/app/db/models.py:24`。
- 前端包描述为 `Novel Forege`，homepage 仍是 electron-vite 模板地址，见 `frontend/package.json:5`、`frontend/package.json:8`。
- `postbuild:web` 引用的脚本未在当前扫描中发现，见 `frontend/package.json:22`。
- Web/NPM 反代场景下，集合接口如果少尾斜杠会触发 FastAPI 307，并可能跳转到不带外部端口的地址。当前已修复项目列表和提示词列表调用，但后续新增集合接口仍需检查。

## 8. 关键风险清单

### 8.1 数据迁移风险

启动补列只适合安全追加列，不适合真实生产迁移。后续涉及 Schema 演进时，必须使用 Alembic 或显式迁移脚本。相关证据见 `backend/app/core/startup.py:54` 到 `backend/app/core/startup.py:60`。

### 8.2 工作流运行一致性风险

工作流运行控制、暂停、取消、并发槽在进程内，见 `backend/app/services/workflow/engine/runtime.py:31` 到 `backend/app/services/workflow/engine/runtime.py:40`。防抖也在进程内，见 `backend/app/services/workflow/triggers.py:21`。单进程本地应用可以接受；服务化、多进程或崩溃恢复场景风险较高。

### 8.3 事件失败不可见风险

事件处理器异常被吞掉并记录日志，见 `backend/app/core/events.py:69` 到 `backend/app/core/events.py:73`。这保持了主流程成功，但会让联动失败不易被用户感知。

### 8.4 API Key 安全风险

`LLMConfig.api_key` 是普通字符串字段，见 `backend/app/db/models.py:24`。如果产品承诺安全存储，应统一后端加密、Electron Keytar 和导入导出策略。

### 8.5 CORS 与 CSP 风险

后端 CORS 默认 `*`，见 `backend/app/core/config.py:163`。Electron CSP 连接源较窄，见 `frontend/src/main/index.ts:47` 到 `frontend/src/main/index.ts:52`；Web CSP 又放宽为 `connect-src *`，见 `frontend/vite.config.web.ts:36` 到 `frontend/vite.config.web.ts:37`。部署前必须按目标形态重审。

### 8.6 前端复杂度风险

`Editor.vue`、`useAssistantStore`、`useCardStore`、工作流组件承载的职责较多。工程规则要求超大组件持续拆分，见 `rules/novelforge-engineering-rule.md:118` 到 `rules/novelforge-engineering-rule.md:121`。

### 8.7 依赖可复现风险

后端 `requirements.txt` 基本未 pin 版本，见 `backend/requirements.txt:1` 到 `backend/requirements.txt:15`。后续如果要做发布或 CI，应考虑锁定版本或生成 lock 文件。

### 8.8 Web 反代路径风险

前端部署到 `https://novel.curdfu.top:18940` 这类带端口反代入口时，必须确保 API 请求保持同源同端口。运行时配置通过 `novelforge-config.js` 注入 `window.location.origin`，见 `scripts/deploy-web.ps1:612`；前端读取该配置见 `frontend/src/renderer/src/api/request.ts:28` 到 `frontend/src/renderer/src/api/request.ts:31`。若接口路径触发 307，浏览器可能访问不带 `:18940` 的错误地址。

## 10. 推荐优先级路线

### P0：修改前必须遵守

- 工作流代码任何写回必须 parse/validate，通过后再应用。
- 不要跳过后端 schema 直接在前端复制长期类型。
- 不要做无关格式化和大范围重命名。
- 涉及数据库模型变化时，不要只依赖启动补列。

### P1：短期高收益修复

- 修复或移除 `frontend/package.json:22` 中缺失的 `postbuild:web` 脚本引用。
- 统一知识图谱 provider 文档与代码命名。
- 明确 API Key 存储策略：明文 DB、后端加密、Electron Keytar 三者选一个主轨。
- 收敛 `useCardStore` 中直接 fetch/SSE/轮询逻辑到统一服务。
- 为后端依赖提供更可复现的版本约束。
- 建立 API 路径尾斜杠检查，防止 Web 反代下再次触发 307 丢端口。

### P2：中期架构改善

- 引入正式 Alembic 迁移流程。
- 将工作流运行控制的关键状态落库或抽象为可替换 runtime。
- 引入 vue-router 替代手写 hash 路由。
- 拆分 `Editor.vue` 和过重 Store。
- 统一前端流式请求、取消、错误处理和 loading 状态机。

### P3：长期演进

- 为工作流运行建立更完整的可观测性：事件日志、错误提示、运行追踪。
- 建立后端 API 契约生成和前端类型同步的 CI 检查。
- 按 Electron/Web/服务化部署分别整理安全策略：CORS、CSP、密钥、更新源。
- 为 AI provider 和 transport 增加契约测试，降低 LangChain 版本漂移影响。

## 11. 修改检查清单

后续每次改动前建议按以下顺序确认：

1. 是否触及数据库模型？如果是，是否需要迁移而不是启动补列。
2. 是否触及工作流代码？如果是，是否完成 parse/validate/run 闭环。
3. 是否触及跨领域联动？如果是，是否通过事件或明确协调层实现。
4. 是否触及 API 契约？如果是，是否更新 OpenAPI 生成类型。
5. 是否触及 AI Key 或模型配置？如果是，是否符合统一密钥策略。
6. 是否触及前端流式请求？如果是，是否具备明确状态机和取消路径。
7. 是否触及大型组件？如果是，是否能局部抽离而不是继续堆叠。
8. 是否新增配置？如果是，是否进入统一配置对象和 `.env.example`。
9. 是否影响 Electron/Web 双模式？如果是，是否同时检查 CSP、BASE_URL、Vite 代理。
10. 是否新增或修改集合 API？如果是，前端路径是否与后端尾斜杠声明一致，避免 307。
11. 是否有验证闭环？后端至少接口/流程验证，前端至少关键路径交互验证，工作流至少 parse/validate/run。
